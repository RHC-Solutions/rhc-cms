# SEO Audit — rhcsolutions.com (2026-05-25)

Fresh end-to-end SEO audit one week after the [2026-05-18 audit](./AUDIT_SEO_2026-05-18.md) (+ its 2026-05-19 follow-up). Scope: live HTML, code paths, CMS data coverage, redirects, status codes, structured data, indexability. Conducted against the running PM2 instance on `web01:3001`.

## Executive summary

The site's SEO machinery is in **strong shape** — the prior audits closed most of the surface-level work. Three new critical or high-impact issues surfaced in this pass:

1. **Sitemap leaks the literal placeholder `/lp/[slug]`** — Googlebot will hit it and 404. Caused by `getStaticRoutes()` in [src/app/sitemap.ts](../src/app/sitemap.ts) not filtering Next.js dynamic-segment directories.
2. **Every sitemap `<lastmod>` is 3+ months stale** (`2026-02-18T15:55:20.000Z`) because [scripts/sync-seo-from-json.mjs](../scripts/sync-seo-from-json.mjs) updates the `seo` column but doesn't bump `updatedAt`. Result: when SEO fields were rewritten on 2026-05-19, no recrawl signal was emitted to Google.
3. **4 published `/lp/*` landing pages** have empty `seo.metaDescription` and short `seo.metaTitle` (26–37 chars) in [cms-data/landing-pages.json](../cms-data/landing-pages.json). Live rendering falls back to hero content (which works) and forces `noindex,nofollow` defensively (also correct), but the editor-facing SEO fields are dead-feeling — same antipattern as the `seo.json.robotsTxt` and `page.seo.keywords` issues from the 2026-05-18 audit.

Everything else (titles, descriptions, canonicals, JSON-LD coverage, redirects, 404s, security headers, OG image) is healthy. Detailed findings below.

## B. Critical issues (new)

### B1. Sitemap leaks `/lp/[slug]` as a literal URL

**Evidence:** `curl http://localhost:3001/sitemap.xml | grep '<loc>'` returns 23 URLs, the last being:

```
https://rhcsolutions.com/lp/[slug]
```

**Cause:** [src/app/sitemap.ts:10-31](../src/app/sitemap.ts#L10) `getStaticRoutes()` walks `src/app/` and adds every directory that contains a `page.tsx`. It only filters dirs starting with `(`, `_`, or named `api`/`admin`. Next.js dynamic-segment dirs (`[slug]`, `[...slug]`, `[[...slug]]`) all start with `[` and slip through.

**Impact:** Googlebot will request `https://rhcsolutions.com/lp/[slug]`, get 404, mark the sitemap as containing invalid URLs (small trust penalty), and crawl budget is wasted.

**Fix:** In `getStaticRoutes()`, add `&& !entry.name.startsWith('[')` to the inclusion test. One-line change in [src/app/sitemap.ts:18](../src/app/sitemap.ts#L18).

### B2. Sitemap `<lastmod>` is 3+ months stale across the entire site

**Evidence:** Every URL in the live sitemap has `<lastmod>2026-02-18T15:55:20.000Z</lastmod>` even though service-page meta titles, descriptions, and JSON-LD wiring all changed on 2026-05-18 and 2026-05-19.

**Cause:** [scripts/sync-seo-from-json.mjs:16-37](../scripts/sync-seo-from-json.mjs#L16) does `UPDATE pages SET seo = ? WHERE id = ?` — it touches only the `seo` column. `updatedAt` is never set, so `cmsDb.getPages()` reads back the unchanged column and the sitemap builder ([src/app/sitemap.ts:61](../src/app/sitemap.ts#L61)) emits the original 2026-02-18 timestamp.

Additionally, three pages (`/services/virtual-office`, `/clients`, `/services`) have `updatedAt` of 2025-12-31 — even older. Those have never been edited via the admin UI; only via the JSON-sync path.

**Impact:** Google's recrawl scheduling for a URL is heavily influenced by `<lastmod>` deltas. After the May 2026 SEO rewrite, no recrawl was prompted; the new titles/descriptions may not be indexed yet.

**Fix:** Two parts:
1. In [scripts/sync-seo-from-json.mjs:16](../scripts/sync-seo-from-json.mjs#L16), change the UPDATE to also set `updatedAt = ?` with `new Date().toISOString()`.
2. Run the script once now to bump `updatedAt` to `2026-05-25` for all 22 pages, then trigger Cloudflare to purge `/sitemap.xml` so the next GSC sitemap-fetch picks it up.

## C. High-priority issues

### C1. Landing pages have empty CMS-level SEO descriptions

**Evidence:** All 4 published landing pages in [cms-data/landing-pages.json](../cms-data/landing-pages.json) have `seo.metaDescription = ''` (descLen = 0) and `seo.metaTitle` of 26–37 chars:

| Slug | titleLen | descLen |
|---|---|---|
| `cloud-migration-q2` | 26 | 0 |
| `it-consulting-that-drives-results` | 33 | 0 |
| `move-to-the-cloud-without-the-chaos` | 37 | 0 |
| `fractional-cio-and-ciso-on-demand` | 35 | 0 |

Live rendering still emits a usable `<meta name="description">` because [src/app/lp/[slug]/page.tsx:13-30](../src/app/lp/[slug]/page.tsx#L13) falls back to the hero subtitle (verified: `/lp/cloud-migration-q2` emits "AWS, Azure and GCP — designed, migrated and managed by senior engineers."). And `noindex, nofollow` is correctly forced regardless of the CMS value, which is the right default for marketing LPs.

**Impact:** Low on direct SEO (LPs are noindex) but the editor experience repeats the antipattern called out in the 2026-05-18 audit (admin fields that don't appear to do anything). When a future LP needs to be indexable (e.g. a high-quality content LP for organic), editors will set the SEO fields, the page will still render `noindex`, and they'll have no way to discover why.

**Fix:** Either (a) have `generateMetadata()` honor `seo.noindex` from `landing-pages.json` so editors can opt a single LP into indexability when warranted, or (b) hide the SEO panel for LPs entirely in the admin UI. Pick one — current state misleads.

### C2. `seo.googleSearchConsoleVerification` is still empty (carried over)

**Evidence:** `cms-data/seo.json.googleSearchConsoleVerification` is the empty string. The wiring (server-rendered `<meta name="google-site-verification">` in `layout.tsx`) is correct — verified by tracing the code — but the value is empty, so no tag is emitted, so the site isn't verified in Search Console via the meta-tag method.

**Impact:** Cannot submit/inspect URLs in GSC, can't see Performance/Coverage reports, can't request manual recrawl after fixing B2.

**Fix:** Manual — admin pastes the GSC verification code into `/admin/seo` → SEO settings page → Verification section.

### C3. FAQPage schema still not implemented

**Evidence:** The 2026-05-18 audit listed FAQPage as missing. Grepping JsonLd.tsx confirms no `faqLd()` builder exists. The site has FAQ-shaped content scattered across `/contact` (booking flow questions), service pages, and `/careers` — none surface as FAQ rich results.

**Impact:** Missed rich-result eligibility. FAQ rich results have been deprioritized by Google since 2023 but still surface for sites with strong authority and clear Q&A markup. Low-effort win.

**Fix:** Add `faqLd({ questions: [{q, a}, ...] })` to [src/components/JsonLd.tsx](../src/components/JsonLd.tsx). Add an FAQ CMS block type to [src/components/cms/BlockRenderer.tsx](../src/components/cms/BlockRenderer.tsx) (renders `<details>/<summary>` for accessibility AND a sibling FAQPage JSON-LD script). Author 3–5 FAQs per service page.

## D. Medium-priority issues

### D1. Sitemap doesn't filter pages with `seo.noindex: true`

**Evidence:** `bucket(slug)` and the page loop in [src/app/sitemap.ts:55-65](../src/app/sitemap.ts#L55) include every published page regardless of `seo.noindex`. Currently no pages have `noindex: true` so this is latent — but the moment an editor uses the (now-wired) `seo.noindex` shortcut, the page will both render `noindex` AND appear in the sitemap, which Google will flag as a soft conflict in GSC's Coverage report.

**Impact:** Future-proofing. Cheap to fix preemptively.

**Fix:** Skip pages where `JSON.parse(p.seo || '{}').noindex === true` (or where `robots.index === false`) in the sitemap loop.

### D2. Service-page content is still thin

**Evidence:** Phase 6 / #17 of the 2026-05-18 audit remains deferred — service pages are 212–289 words. Google's Helpful Content classifier penalizes thin pages on competitive queries like "cyber security consulting", "CISO as a service".

**Impact:** Probably the single largest remaining drag on organic traffic. The technical SEO is now ~95% complete; content depth is the limiting factor.

**Fix:** Per-page expansion to 600–1000 words: problem framing → RHC's approach → 1–2 outcome bullets → FAQ section (synergizes with C3) → CTA. Human-authored. Tracked but not actionable for an AI pass.

### D3. Career sub-page meta titles are 38–51 chars (short)

**Evidence:** From the CMS-data audit:

| Slug | titleLen |
|---|---|
| `/careers/global-team` | 38 |
| `/careers/work-life-balance` | 44 |
| `/careers/continuous-learning` | 46 |
| `/careers/competitive-compensation` | 51 |

The 2026-05-19 follow-up rewrote 8 service titles and 4 career descriptions, but not the career sub-page titles themselves.

**Fix:** Rewrite to 52–60 chars adding the "| RHC Solutions" suffix plus a context word (e.g., `Global Distributed Team — Remote-First Careers | RHC Solutions`).

### D4. Legal-page titles in hardcoded `Metadata` are 29–32 chars (short)

**Evidence:** `src/app/{privacy,terms,cookies}/page.tsx` each export `Metadata` with titles like `"Privacy Policy | RHC Solutions"` (31 chars). Below the SERP-target 50–60.

**Impact:** Low — legal pages rarely get organic traffic — but cheap to expand for consistency.

**Fix:** Add a brief context phrase: `"Privacy Policy — How RHC Solutions Protects Your Data"` (53 chars).

## E. Low-priority / observations

- **E1. Per-page `keywords` field is universally empty.** The plumbing (added in 2026-05-18 Phase 1) works, but no page sets it. Keywords have no direct ranking signal since ~2009; the value here would be admin-team focus discipline, not Google ranking. Skip unless used as an editorial process tool.

- **E2. `/api/og` cache hit rate is effectively zero.** Title + description vary per page, so each call generates a fresh image. The current `Cache-Control: public, max-age=3600, s-maxage=86400, stale-while-revalidate=604800` is correct, but with 22 unique pages × bots there's a steady tail of edge function invocations. Adding `experimental.dynamicIO` or an LRU keyed on `(title, description, tag)` in the route handler would help — but only if invocation cost becomes a problem.

- **E3. GA4 / GTM / Ahrefs / Hotjar / ContentSquare** — five trackers still loaded. Each costs INP. The 2026-05-18 audit punted this to "pull CrUX field data before pruning." Still the right call. Worth re-checking once GSC is verified (C2) and CrUX is accessible.

- **E4. `bucket(slug)` doesn't classify `/lp/*` paths.** Falls through to the default `{priority: 0.5, changeFrequency: 'monthly'}` — but moot once B1 is fixed (LP pages should not be in the sitemap at all because they're noindex).

- **E5. JobPosting JSON-LD uses `validThrough = postedDate + 90 days` when not set explicitly** ([src/components/JsonLd.tsx:155](../src/components/JsonLd.tsx#L155)). This means stale jobs in `jobs.json` get expired-but-still-emitted JSON-LD after 90 days, which Google flags as "Posting Expired" in the Rich Results test. Either explicitly set `validThrough` in each job or trim `jobs.json` regularly.

- **E6. The `[slug]` directory issue (B1) also catches future dynamic routes.** If anyone adds e.g. `src/app/case-studies/[slug]/page.tsx`, the same bug will leak `/case-studies/[slug]`. The fix in B1 covers all future dynamic routes too.
