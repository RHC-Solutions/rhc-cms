# SEO Audit — rhcsolutions.com (2026-05-18)

Deep SEO audit of the live Next.js 16 (App Router) site. Companion to the broader [AUDIT_DESIGN_SEO_UX_2026-05-11.md](./AUDIT_DESIGN_SEO_UX_2026-05-11.md); this doc is the focused SEO follow-up and feeds the improvement plan at the bottom.

## Executive summary

The technical foundation is **solid but incomplete**. Metadata, sitemap, robots, JSON-LD basics, font/image config, CSP, and consent handling are all wired correctly. The major gaps are:

1. **Thin content** across most pages (19 of 22 pages under 300 words).
2. **CMS-level SEO fields that exist but are never consumed by the renderer** — `page.seo.robots`, `page.seo.keywords`, `seo.googleSearchConsoleVerification`, `seo.json.robotsTxt`.
3. **Missing structured data** for the service-oriented business model (no `Service`, `LocalBusiness`, `FAQPage` schemas).
4. **7 orphaned pages** not linked from header, footer, or any card block.
5. **No www→apex redirect** — full-site duplicate-content risk if both hostnames resolve.

Phased by leverage-vs-effort. Each item is independently shippable.

### Phase 1 — Code unlocks (1 PR, ~1–2 hours)
High leverage, low risk. All in code, no content writing.

5. Add `priority={true}` to the first image block per page in [BlockRenderer.tsx](../src/components/cms/BlockRenderer.tsx).
6. Add www→apex 301 in [middleware.ts](../middleware.ts) (single early-return check).

### Phase 2 — Structured data (1 PR, ~2 hours)
Pure additive markup. Big rich-result eligibility win.

### Phase 3 — Dynamic OG images (1 PR, ~1 hour)

### Phase 4 — Sitemap & URL hygiene (small PR)

14. Tighten the `/web-check` X-Robots-Tag match in [middleware.ts:48](../middleware.ts#L48).

### Phase 5 — Internal linking (CMS-data only)

    - Add IT Consulting, CIO-aaS, CISO-aaS cards to the `/services` page's cards block in [cms-data/pages.json](../cms-data/pages.json).

## 2026-05-19 follow-up audit

Fresh pass over the surface after Phases 1–6 landed. Findings the original audit didn't cover, ranked by impact, plus what shipped.

### Critical — fixed

**N1 — `GET /api/cms/seo` was leaking secrets through the public CDN cache.**
The endpoint was in `publicApiEndpoints` (auth bypass on GET) and returned the entire `seo.json`, which on this server stores `ahrefsApiKey`, `ipinfoToken`, `ahrefsId` (admin email) and `updatedBy` alongside the public GA/GTM IDs. The `/api/:path*` rule in `next.config.mjs` applies `Cache-Control: public, max-age=300, s-maxage=600` to every API response, so an unauthenticated `curl https://rhcsolutions.com/api/cms/seo` returned the secret keys and could be cached at the Cloudflare edge for 10 minutes.

The 2026-05-13 SECURITY_REMEDIATION addressed git-history exposure of `seo.json` but missed the runtime HTTP path. Fixed by **(a)** removing `/api/cms/seo` from `publicApiEndpoints` in `middleware.ts`, **(b)** adding a `scrubForPublic()` step inside the GET handler that drops `ahrefsId`/`ahrefsApiKey`/`ipinfoToken`/`updatedBy` for non-admin/editor tokens, and **(c)** explicitly setting `Cache-Control: private, no-store` on the response so the privileged admin variant cannot be served from a shared cache. Only client-side caller of `/api/cms/seo` was the `GoogleAnalytics` fallback fetch, which already degrades gracefully (SSR prop + env var) when the call 401s.

### High — fixed

**N2 — Bogus `Link: rel=preload` middleware header.**
`middleware.ts` emitted `Link: </_next/static/css>; rel=preload; as=style` on every HTML response, but `/_next/static/css` is a directory — Next emits hashed filenames (e.g. `/_next/static/css/<hash>.css`), so every browser followed the preload and got a 404. Removed; replaced with a comment explaining why a generic CSS preload is wrong here.

**N3 — Service-page meta titles were 32–45 chars (under SERP target).** All 8 `/services/*` titles rewritten to 48–59 chars with a primary-keyword + brand suffix (`Cyber Security Services & Consulting | RHC Solutions`, `CIO as a Service — Fractional IT Leadership | RHC Solutions`, etc.). Stored in `cms-data/pages.json` and synced into `cms.db` via `scripts/sync-seo-from-json.mjs`.

**N4 — Career sub-page meta descriptions were 126–139 chars.** All 4 `/careers/*` descriptions padded to 151–157 chars with concrete detail (numbers of offices, types of training, etc.).

### Medium — fixed

**N5 — `WebSite` JSON-LD missing `description` and `inLanguage`.** Added both to `websiteLd()` in `src/components/JsonLd.tsx`, sourcing the description from `settings.brand.valueProp` with sensible fallbacks. Verified live in `/index.html`.

**N6 — Deprecated/harmful security headers shipped on every response.** Removed `X-XSS-Protection: 1; mode=block` (deprecated by all major browsers; the `mode=block` variant has been shown to *introduce* reflected-XSS class bugs in some browser combos — CSP is the modern replacement and was already configured) and `Expect-CT` (retired by Chromium in 2023, silently ignored). Changed `Referrer-Policy` from `same-origin` (stripped the Referer header on every outbound link, hiding our referrals from clients/partners) to `strict-origin-when-cross-origin` (the W3C-recommended default — sends the origin but not the path, preserving referral attribution while still protecting URL privacy).

### Intentionally not changed

- **HSTS at origin** — Cloudflare owns TLS termination and applies `Strict-Transport-Security` at the edge; duplicating at origin risks locking out a future apex-without-TLS migration path. Documented in the `security-headers.mjs` comment.
- **Legal-page meta** — found in pages.json as 71–85 chars but those rows are dead data: `/privacy`, `/terms`, and `/cookies` are hard-coded routes (`src/app/{privacy,terms,cookies}/page.tsx`) with their own inline `Metadata` constants that are already 140–178 chars. The renderer never consults pages.json for these slugs. (`pages.json` edits to those rows were left in place so the JSON snapshot at least matches the live values within ±5 chars.)
- **`googleSearchConsoleVerification` value** — still empty in `seo.json`. The wiring is correct; the admin just hasn't pasted the GSC code into `/admin/seo`. Manual task.
- **Content expansion (Phase 6 #17 / #19)** — still needs human-written copy.

### Verification

- `npx tsc --noEmit` — clean.
- Grepped post-build `/services/cyber-security.html`, `/privacy.html`, `/careers/global-team.html`, `/index.html` — all new titles, descriptions, and JSON-LD fields are present in the prerendered output.
