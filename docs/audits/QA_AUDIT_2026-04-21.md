## 5. Security Findings

### 5.1 🔴 CRITICAL — Secrets committed in `cms-data/`

`git ls-files` confirms these files are **tracked in version control**:

- [cms-data/users.json](cms-data/users.json) — contains `passwordHash` (bcrypt) and a plaintext `totpSecret` (`DNNRIX7RBEMBABDPLLQQF6GD7OQGQQ5R`). Anyone with repo access can clone the TOTP seed and bypass MFA.
- [cms-data/seo.json](cms-data/seo.json) — contains a real-looking `ahrefsApiKey` and `ipinfoToken` (`e7dda13207bb37`).

**Remediation (must do all three):**
1. **Rotate** the Ahrefs key, the IPinfo token, and the admin password / TOTP seed *now* — assume they are public.
2. Move secrets out of `cms-data/*.json` into env vars (or an encrypted store). Keep only non-sensitive config in tracked JSON.
3. Add `cms-data/users.json`, `cms-data/seo.json`, and any other secret-bearing file to `.gitignore`, then **purge from history** with `git filter-repo` or BFG before pushing.

### 5.2 🔴 HIGH — Hardcoded NEXTAUTH_SECRET fallback

Three places fall back to the literal string `'your-secret-key-change-in-production'`:

- [src/proxy.ts](src/proxy.ts#L16)
- [src/lib/auth/check.ts](src/lib/auth/check.ts#L18)
- [src/lib/auth/config.ts](src/lib/auth/config.ts#L245)

If `NEXTAUTH_SECRET` is ever unset in production, JWTs are signed with a public, predictable secret → trivial session forgery / admin takeover.

**Fix:** Throw on missing `NEXTAUTH_SECRET` at boot instead of silently falling back:
```ts
const secret = process.env.NEXTAUTH_SECRET;
if (!secret) throw new Error('NEXTAUTH_SECRET is required');
```

### 5.3 🟠 HIGH — `next/image` `remotePatterns: '**'`

[next.config.mjs](next.config.mjs#L13-L18) allows **any HTTPS host** as an image source. This is the exact pattern called out by the Next.js Image Optimizer DoS advisory (GHSA-9g9p-9gw9-jx7f) — an attacker can request `/_next/image?url=https://attacker/...` to make the server fetch and re-encode arbitrary remote payloads.

**Fix:** Restrict to the small set of hostnames you actually serve from (your CDN, S3 bucket, Cloudflare Images, etc.).

### 5.4 🟠 MEDIUM — CSP weakened by `unsafe-inline` + `unsafe-eval`

[middleware.ts](middleware.ts) sets a CSP that allows `'unsafe-inline' 'unsafe-eval'` for `script-src`. Combined with the multiple `dangerouslySetInnerHTML` sites (see 5.5), this nullifies most XSS protection.

The static [_headers](_headers#L11) file ships a stricter CSP without `'unsafe-eval'`. Cloudflare Pages will use `_headers`; Node-hosted deployments (per [package.json](package.json#L8) `next start`) will use middleware. **The two are out of sync.**

**Fix:**
1. Pick one source of truth (recommend middleware so it follows the deployment).
2. Remove `'unsafe-eval'`.
3. Move inline `<script>`s in [src/app/layout.tsx](src/app/layout.tsx#L119) to nonced scripts and add `'nonce-…'` to CSP, or to external files.

### 5.5 🟡 MEDIUM — `dangerouslySetInnerHTML` on CMS-controlled HTML

[src/components/cms/BlockRenderer.tsx](src/components/cms/BlockRenderer.tsx#L443) renders the `richtext` block's `props.html` directly. The CMS is admin-only, so this is XSS-by-admin rather than by-anonymous, but:

- An admin compromise (cf. 5.1, 5.2) becomes a stored-XSS that hits every visitor.
- There is no sanitization layer.

**Fix:** Run `html` through a sanitizer (DOMPurify with jsdom on the server, or `sanitize-html`) before rendering.

### 5.6 🟡 MEDIUM — `/api/apply` upload endpoint

[src/app/api/apply/route.ts](src/app/api/apply/route.ts) accepts a `resume` file from a public form and forwards it to Telegram. Found:

- ✅ Cloudflare Turnstile is verified.
- ✅ Disposable-email + phone format checks.
- ❌ **No file size cap** in the handler — relies on Next's default body limit (which is large).
- ❌ **No MIME / extension validation** — any file type is forwarded.
- ❌ **No rate limiting** beyond Turnstile (one-shot tokens, but token re-issuance is cheap).
- ❌ Application data is appended to a JSON file with no atomicity guarantee — concurrent submissions can lose writes.

**Fix:** Cap `resume.size` (e.g. 5 MB), allowlist `application/pdf`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, add IP-based rate limiting (Cloudflare WAF rule or a small in-process limiter), and serialize writes (rename-temp pattern or move to SQLite — a `better-sqlite3` is already installed).

### 5.7 🟢 LOW — Middleware bypass header check is good

[middleware.ts](middleware.ts) explicitly rejects requests carrying `x-middleware-subrequest` (the Next.js auth-bypass CVE). Keep this.
